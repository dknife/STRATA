\section{실험 개요}
%======================================================================

본 보고서는 11종의 APSP 기법을 6개 그래프에서 비교한 결과를 기록한다. 이전 보고서(4월 1일) 대비 TG2 D-STORM-CUDA의 footprint 체크를 non-atomic에서 \textbf{guard+CAS}로 수정하여 race condition을 제거하고, 성능 변화를 재측정하였다.

\subsection{TG2 커널 수정 사항}
\begin{tcolorbox}[colback=blue!3, colframe=blue!50!black, title=guard+CAS 패턴 (2026-04-02 적용)]
기존 non-atomic 체크는 두 스레드가 같은 $F[i \cdot n + k]$를 동시에 0으로 읽어 중복 출력을 발생시킬 수 있었다. 대규모 그래프에서 중복 frontier가 출력 버퍼($n^2$)를 초과하면 \textbf{메모리 오버플로우} 위험이 있다.

\medskip
\textbf{수정}: non-atomic read guard + \texttt{atomicCAS}로 교체.
\begin{verbatim}
  if (F[idx] == 0) {                  // non-atomic guard: 이미 1이면 skip
      if (atomicCAS(&F[idx], 0, 1) == 0) {  // CAS로 원자적 갱신
          int pos = atomicAdd(out_count, 1);
          out_row[pos] = i;
          out_col[pos] = k;
      }
  }
\end{verbatim}
\begin{itemize}[leftmargin=1.4em]
  \item \textbf{정확성}: CAS가 중복 출력을 원천 차단 (모든 그래프 PASS).
  \item \textbf{성능}: non-atomic read guard가 이미 방문한 셀($F=1$)에서 atomic을 건너뛰어 contention 회피. Facebook(n=4,039)에서 \texttt{atomicExch} 대비 \textbf{38\% 빠름}, non-atomic 대비 동등.
\end{itemize}
\end{tcolorbox}

\subsection{프로토콜}
\begin{itemize}[leftmargin=1.4em]
  \item \textbf{측정 횟수}: 3회 실행, 최소값 보고.
  \item \textbf{GPU warmup}: GPU 기법은 1회 사전 실행 후 측정 시작.
  \item \textbf{정확성 검증}: 모든 기법의 거리 행렬을 원소별 비교 (\textbf{11종 전부 PASS}).
  \item \textbf{총 측정점}: 11 methods $\times$ 6 datasets = 66건 (TC3는 BC5와의 GraphBLAS 초기화 충돌로 별도 프로세스에서 측정).
\end{itemize}

\section{벤치마킹 대상 기법}
%======================================================================

\begin{tcolorbox}[colback=gray!3, colframe=gray!60!black, title=핵심 연산 용어 해설, breakable]
\begin{description}[style=nextline, leftmargin=1.2em, labelwidth=0em, font=\bfseries]

\item[Edge-wise 연산]
행렬곱을 사용하지 않고, 각 정점 $u$의 이웃 목록을 순회하며 도달 가능성 행 $R[u,:]$를 직접 갱신하는 방식. I-AORM(BC3)에서 사용. Python 루프로 구현되어 느리지만, 알고리즘 구조가 단순하여 GPU에서 부활한다 (TG2, BG1의 CSR 이웃 순회가 본질적으로 edge-wise).

\item[BLAS matmul (Basic Linear Algebra Subprograms)]
CPU에서 고도로 최적화된 행렬곱 라이브러리. NumPy의 \texttt{A.dot(B)}가 내부적으로 호출하며, SIMD 벡터 명령어와 캐시 블로킹으로 $O(n^3)$ 연산을 가속한다. M-AORM(BC4)과 D-STORM-NumpyBLAS(TC2)에서 dense 행렬곱의 핵심 엔진.

\item[GrB\_vxm (GraphBLAS Vector-Matrix Multiply)]
GraphBLAS API의 벡터--행렬 곱 연산. 하나의 소스 벡터(frontier)를 인접 행렬과 곱하여 1-hop 이웃을 확장한다. GB-bfs(BC5)에서 소스별 BFS의 핵심 연산으로, $n$회 반복 호출하여 APSP를 구한다.

\item[SpMM (Sparse Matrix--Matrix Multiply)]
희소 행렬 간의 행렬곱. 0이 아닌 원소만 처리하여 dense matmul의 $O(n^3)$을 실제 비영 원소 수에 비례하는 비용으로 줄인다. D-STORM-SpMM-Cython(TC1)에서 SciPy CSR SpMM을, 기존 GPU-Sparse에서 cuSPARSE SpMM을 사용. D-STORM의 반복적 frontier 확장의 핵심이나, 이미 방문한 쌍까지 계산하는 구조적 비효율이 존재한다.

\item[Cython]
Python 코드를 C로 컴파일하는 트랜스파일러. TC1에서 SpMM 출력의 COO 스캔과 footprint 갱신을 단일 C 패스로 융합(fused prune)하여, Python 루프 대비 수십 배 가속한다. 3단계 sparse 연산(booleanize $\to$ prune $\to$ footprint update)을 1-pass로 통합하는 핵심 최적화.

\item[NumPy BLAS]
NumPy가 제공하는 dense 행렬 연산 인터페이스. 내부적으로 BLAS/LAPACK을 호출하며, TC2에서 $A \cdot R_k$ 행렬곱과 Heaviside pruning에 사용. 희소성을 활용하지 못해 고지름 그래프에서 반복 횟수에 비례하여 성능이 저하된다.

\item[GrB\_mxm (GraphBLAS Matrix-Matrix Multiply)]
GraphBLAS API의 행렬--행렬 곱 연산. 사용자 정의 세미링(semiring)을 지정할 수 있어, TC3에서는 LOR\_LAND 세미링과 구조적 보수 마스크(complement mask)를 결합하여 D-STORM의 frontier 확장 + pruning을 단일 호출로 수행한다.

\item[cuBLAS matmul]
NVIDIA GPU에서 실행되는 BLAS 행렬곱 라이브러리. FP32 Tensor Core를 활용하여 dense 행렬곱을 GPU에서 수행한다. TG1에서 $A \cdot R_k$를 GPU dense matmul로 처리. CPU BLAS 대비 수배 빠르나, $O(n^3)$ 구조는 동일하여 Tier~1(TG2/BG1) 대비 느리다.

\item[CUDA CSR 직접 확장]
cuSPARSE SpMM을 제거하고, frontier의 각 엔트리 $(i,j)$를 CUDA 스레드에 할당하여 $j$의 CSR 이웃을 직접 순회하는 커스텀 커널. TG2에서 사용. SpMM의 세 가지 병목(불필요한 곱 계산, 중간 행렬 생성, 별도 prune 단계)을 제거하여 6--18배 개선을 달성한다.

\item[guard+CAS (Compare-And-Swap)]
CUDA 커널에서 공유 데이터의 원자적 갱신을 위한 2단계 패턴. 먼저 \textbf{non-atomic read}로 이미 1인 셀을 빠르게 건너뛴 뒤(guard), 0인 셀에 대해서만 \texttt{atomicCAS(\&F[idx], 0, 1)}로 원자적 $0 \to 1$ 전이를 수행한다. 순수 atomic(\texttt{atomicExch}) 대비 contention을 대폭 줄이면서 중복 출력을 완전히 방지한다. TG2의 footprint 갱신에 적용.

\end{description}
\end{tcolorbox}

\subsection{D-STORM 최적화 극한: per-source BFS로의 수렴}

D-STORM의 SpMM 병목을 순차적으로 제거하면 per-source BFS에 수렴함을 실험적으로 확인하였다.

\begin{table}[H]\centering
\caption{D-STORM 최적화 경로 (Facebook 기준)}
\label{tab:convergence}
\small
\begin{tabular}{@{}llrr@{}}
\toprule
\textbf{단계} & \textbf{변경 사항} & \textbf{시간 (s)} & \textbf{개선} \\
\midrule
GPU-Sparse & cuSPARSE SpMM + sparse prune & 0.192 & 1.0$\times$ \\
DirectExpand (atomicExch) & SpMM $\to$ CSR 직접 확장 & 0.041 & 4.7$\times$ \\
DirectExpand (guard+CAS) & atomicExch $\to$ guard+CAS & 0.030 & 6.4$\times$ \\
BG1 GPU-PerSrc-BFS & dist 배열 1개로 통합, 단일 launch & \textbf{0.019} & \textbf{10.1$\times$} \\
\bottomrule
\end{tabular}
\end{table}

\paragraph{구조적 수렴.}
D-STORM의 행렬 대수적 접근은 다음 구조를 강제한다:
\begin{enumerate}[leftmargin=1.4em]
  \item \textbf{Frontier 행렬} ($n \times n$): 전체 소스의 frontier를 하나의 행렬로 통합.
  \item \textbf{Footprint 행렬}: 전체 소스가 공유 $\to$ 쓰기 시 동기화(atomic) 필요.
  \item \textbf{반복 커널 launch}: hop당 1회 $\to$ 지름에 비례하는 오버헤드.
\end{enumerate}

반면 per-source BFS(BG1)는:
\begin{enumerate}[leftmargin=1.4em]
  \item 소스 간 \textbf{완전 독립} $\to$ atomic 불필요.
  \item \texttt{dist} 배열이 frontier/visited/distance를 겸용 $\to$ 메모리 1/3.
  \item \textbf{단일 커널} launch $\to$ 반복 오버헤드 0.
\end{enumerate}

\section{종합 결론}
%======================================================================

\begin{tcolorbox}[colback=green!5, colframe=green!50!black, title=핵심 결론]
\begin{enumerate}[leftmargin=1.4em]
  \item \textbf{GPU-PerSrc-BFS(BG1)가 전체 최고}: SciPy 대비 23--101배. per-source BFS의 CUDA 병렬화가 full APSP에서 가장 효율적.
  \item \textbf{D-STORM-CUDA(TG2)가 D-STORM GPU 최고}: guard+CAS로 정확성을 보장하면서 non-atomic 동등 성능 유지. 소세계($n \approx 2{,}000$)에서 BG1을 18--34\% 상회.
  \item \textbf{guard+CAS 패턴의 검증}: non-atomic read guard가 이미 방문한 셀에서 atomic을 건너뛰어, \texttt{atomicExch} 대비 38\% 빠르면서 정확성 보장. CUDA 커널의 monotonic flag 패턴에 범용 적용 가능.
  \item \textbf{D-STORM 최적화의 극한은 per-source BFS}: SpMM $\to$ 직접 확장 $\to$ guard+CAS로 최적화하면 BG1과 유사한 구조에 수렴. 남은 1.6배 차이는 단일 launch와 메모리 구조의 차이.
  \item \textbf{조건부 운영 영역}:
    \begin{itemize}
      \item GPU 가용 시: BG1 사용 (대규모/고지름 최고), 소세계에서는 TG2도 경쟁력 있음.
      \item CPU 전용: TC1 D-STORM-SpMM-Cython (모든 위상에서 CPU 최고, SciPy 대비 1.0--1.9배).
      \item $k$-hop 구조적 인터페이스 필요: D-STORM (TC1 또는 TG2).
    \end{itemize}
\end{enumerate}
\end{tcolorbox}
