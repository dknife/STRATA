Overview

STORM (Sparse Topology-aware Optimal Reachability Matrix) is a Cython-accelerated framework for incremental all-pairs shortest path (APSP) computation. It extends the AORM framework (IEEE Access, 2021) with sparse matrix redesign, a fused C pruning kernel, GPU acceleration via CUDA, and comprehensive benchmarking against GraphBLAS.

Key Results (Facebook, n=4,039)

  • GPU-Dense (cuBLAS): 0.114s — 128.8× over NetworkX, 17.9× over SciPy (C)
  • D-STORM-Sparse (Cython): 1.093s — 1.87× over SciPy (C), 7.4× over I-AORM
  • Hop-constrained (k=5): 18.3× over NetworkX at 89% accuracy

Why STORM?

Real-world networks are overwhelmingly small-world: Facebook's 721 million users are connected by an average distance of just 4.74 hops. Most applications—traffic prediction, epidemic tracing, influence analysis—need only local distance information (k=3–5 hops), not full APSP. STORM's incremental design provides this information faster than any existing method.

Contributions

  1. Systematic audit: Seven AORM limitations identified via code-paper comparison.
  2. Optimized CPU implementation: Five structural changes + Cython fused pruning kernel → 7.4× over I-AORM, 1.87× over SciPy (C) on Facebook.
  3. GPU acceleration: CUDA port via cuBLAS/cuSPARSE → 10–16× over CPU D-STORM, scaling with graph size.
  4. GraphBLAS baseline: Same algorithm reimplemented with SuiteSparse:GraphBLAS → D-STORM's Cython kernel is 1.6–2.0× faster on small-world graphs.
  5. Topology-aware analysis: Graph diameter governs performance. STORM excels on d≤8, GPU eliminates the high-diameter weakness.
  6. Dynamic support (D-STORM): 22–33× faster edge addition via incremental distance updates.

Algorithm

STORM preserves AORM's iterative reachability framework while introducing five structural changes:

IDAORMSTORMEffect
C1Dense matmul O(n³)Sparse SpMM O(nnz·c̄)Sparse graph acceleration
C2Fixed O(n²)/iternnz-proportional costProgressive sparsity
C3Pre-prune check O(n²)Post-prune nnz=0, O(1)Correct + fast
C4heaviside O(n²)data[:]=1 O(nnz)nnz-only work
C5Sparse F add O(n²)Dense bool F, O(1)/entryFootprint acceleration

The Cython C-extension (_storm_core.pyx) fuses pruning, convergence checking, and footprint update into a single C loop. On Windows, it is built with MSVC 14.44 (VS 2022 Build Tools) via a manual cl+link pipeline.

D-STORM: Dynamic Edge Addition

When edge (u,v) is added: Di,jnew = min(Di,jold, Di,u + 1 + Dv,j). Only 0.008% of pairs are affected per update, yielding 22–33× speedup over full recomputation.

Discussion

Performance Hierarchy (Small-World Graphs)

GPU (0.02–0.2s) ≫ D-STORM CPU (0.2–1.1s) > SciPy (0.3–2.0s) > AORM / GB-frontier (0.4–3.5s) > NetworkX / GB-bfs (1–15s)

When SciPy Wins — and Why GPU Solves It

SciPy outperforms CPU D-STORM for full APSP on Grid (d=88). But GPU-Dense (0.094s) beats SciPy (0.151s) even on Grid. With GPU available, D-STORM has no topology weakness.

Even without GPU: (1) small-world networks dominate practice; (2) hop-constrained STORM is faster for k≤16 even on Grid; (3) SciPy cannot provide incremental queries, intermediate results, or dynamic updates.
